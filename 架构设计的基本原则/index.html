架构设计的基本原则

发表于 2026-02-22 更新于 2026-02-24 分类于 架构基础

引言

在上一篇文章中，我们建立了架构思维的基础，理解了架构的价值。现在，让我们深入探讨架构设计的核心原则。这些原则是优秀架构的基石，无论技术如何变化，这些基本原则始终适用。

本文将介绍架构设计中最重要、最实用的基本原则，帮助你建立坚实的设计基础。

一、SOLID原则：面向对象设计的基石

SOLID原则是Robert C. Martin提出的五个面向对象设计原则的缩写，它们同样适用于架构设计。

1.1 单一职责原则 (Single Responsibility Principle, SRP)

一个类（或模块）应该只有一个引起变化的原因。

实践建议：

  • 每个模块只负责一个明确的业务功能
  • 避免”上帝类”或”万能模块”
  • 通过职责划分提高可维护性

1.2 开闭原则 (Open-Closed Principle, OCP)

软件实体应该对扩展开放，对修改关闭。

实践建议：

  • 通过接口和抽象实现扩展性
  • 使用策略模式、工厂模式等设计模式
  • 避免直接修改现有代码

1.3 里氏替换原则 (Liskov Substitution Principle, LSP)

子类必须能够替换其父类，而不影响程序的正确性。

实践建议：

  • 子类不应该改变父类的行为约定
  • 避免在子类中抛出父类没有声明的异常
  • 保持接口契约的一致性

1.4 接口隔离原则 (Interface Segregation Principle, ISP)

客户端不应该依赖它不需要的接口。

实践建议：

  • 创建小而专一的接口
  • 避免”胖接口”
  • 按需实现接口

1.5 依赖倒置原则 (Dependency Inversion Principle, DIP)

高层模块不应该依赖低层模块，两者都应该依赖抽象。

实践建议：

  • 依赖抽象，而不是具体实现
  • 使用依赖注入
  • 提高代码的可测试性和灵活性

二、DRY原则：避免重复

Don’t Repeat Yourself (不要重复你自己)

2.1 DRY的核心思想

  • 每一处知识在系统中都应该有单一、明确、权威的表示
  • 重复是万恶之源，会导致维护困难

2.2 如何实践DRY原则

  1. 提取公共方法：将重复代码提取为函数或方法
  2. 使用模板：对于相似的代码结构
  3. 创建基类：对于共享的行为
  4. 配置化：将硬编码的值提取为配置

2.3 DRY的误区

  • 不要过度抽象：过早的抽象可能带来不必要的复杂度
  • 区分逻辑重复和偶然重复：有些重复是必要的

三、KISS原则：保持简单

Keep It Simple, Stupid (保持简单，傻瓜)

3.1 简单性的价值

  • 简单的设计更容易理解和维护
  • 减少出错的可能性
  • 提高开发效率

3.2 如何实践KISS原则

  1. 避免过度设计：只解决当前的问题
  2. 选择简单的解决方案：在满足需求的前提下选择最简单的
  3. 清晰的命名：让代码自解释
  4. 减少依赖：最小化外部依赖

四、YAGNI原则：你不需要它

You Aren’t Gonna Need It (你不需要它)

4.1 YAGNI的核心

  • 不要为未来可能的需求编写代码
  • 专注于当前的需求
  • 避免”以防万一”的设计

4.2 如何实践YAGNI

  1. 按需开发：只在需要时添加功能
  2. 重构优于预测：当需求变化时重构代码
  3. 保持代码灵活：但不为不存在的需求设计

五、关注点分离 (Separation of Concerns, SoC)

5.1 SoC的核心思想

  • 将系统分解为不同的部分，每个部分解决一个特定的关注点
  • 降低系统的复杂度

5.2 实践示例

  • MVC模式：模型、视图、控制器分离
  • 分层架构：表现层、业务层、数据层分离
  • 微服务：按业务领域分离服务

六、其他重要原则

6.1 最小惊讶原则 (Principle of Least Astonishment)

  • 系统的行为应该符合用户的预期
  • 避免令人惊讶的设计

6.2 高内聚低耦合 (High Cohesion, Low Coupling)

  • 高内聚：模块内部元素紧密相关
  • 低耦合：模块之间依赖最小化

6.3 失败快速原则 (Fail Fast)

  • 尽早发现并报告错误
  • 避免错误在系统中传播

七、原则的权衡与平衡

7.1 没有银弹

  • 不同原则之间可能存在冲突
  • 需要根据具体场景权衡

7.2 平衡的艺术

场景 优先考虑的原则
快速原型 KISS, YAGNI
长期维护的系统 SOLID, DRY
高性能系统 简单性，避免过度抽象
团队协作项目 一致性，清晰的接口

八、实践指南

8.1 代码审查清单

  • 每个类/模块是否只有一个主要职责？
  • 是否有重复的代码可以提取？
  • 设计是否足够简单？
  • 是否添加了不必要的功能？
  • 关注点是否清晰分离？

8.2 重构时机

  1. 发现重复代码时立即重构
  2. 添加新功能困难时考虑设计问题
  3. 代码难以理解时简化设计
  4. 测试困难时检查耦合度

总结

架构设计原则是指南针，不是铁律。理解这些原则的核心思想比机械地应用它们更重要。在实际工作中，你需要：

  1. 理解原则背后的思想
  2. 根据具体场景灵活应用
  3. 在原则之间找到平衡
  4. 持续学习和反思

记住：好的架构是演进而来的，不是设计出来的。从简单的设计开始，根据需求逐步演进。

思考与实践

思考题

  1. 在你的项目中，哪个原则被违反得最多？造成了什么后果？
  2. SOLID原则中，你认为哪个原则最重要？为什么？
  3. 什么时候应该违反这些原则？

实践任务

  1. 选择一个你熟悉的项目，用本文的原则分析其设计
  2. 找出至少3处可以改进的地方
  3. 制定一个简单的重构计划
