常见的架构模式入门

发表于 2026-02-24 分类于 架构基础

前言

在软件架构的世界里，选择合适的架构模式就像为建筑选择合适的设计方案。不同的业务需求、团队规模和技术栈需要不同的架构模式。本文将介绍几种常见的架构模式，帮助你理解它们的核心思想、适用场景和优缺点。

1. 单体架构 (Monolithic Architecture)

核心思想

将所有功能模块打包在一个应用中，共享同一个代码库、数据库和部署单元。

架构特点

  • 单一代码库
  • 统一部署
  • 共享数据库
  • 进程内通信

优点

  • ✅ 开发简单：技术栈统一，新人上手快
  • ✅ 部署简单：一个包搞定所有
  • ✅ 测试简单：端到端测试容易
  • ✅ 性能优秀：进程内调用，无网络开销

缺点

  • ❌ 扩展困难：只能垂直扩展
  • ❌ 技术栈单一：难以引入新技术
  • ❌ 维护困难：代码库臃肿，耦合度高
  • ❌ 部署风险：一个小改动需要全量部署

适用场景

  • 初创公司 MVP
  • 小型项目（团队<10人）
  • 内部工具
  • 验证性项目

2. 分层架构 (Layered Architecture)

核心思想

按职责将系统划分为多个层次，每层只与相邻层交互。

常见分层

  1. 表现层 (Presentation Layer)：用户界面
  2. 业务层 (Business Layer)：核心业务逻辑
  3. 数据访问层 (Data Access Layer)：数据库操作
  4. 基础设施层 (Infrastructure Layer)：外部服务

变体模式

  • MVC (Model-View-Controller)
  • MVP (Model-View-Presenter)
  • MVVM (Model-View-ViewModel)

优点

  • ✅ 职责清晰：每层有明确职责
  • ✅ 易于维护：修改一层不影响其他层
  • ✅ 可测试性：层间接口明确，便于单元测试

缺点

  • ❌ 性能开销：层间调用有开销
  • ❌ 灵活性差：严格的分层可能限制设计
  • ❌ 过度设计风险：可能增加不必要的抽象

适用场景

  • 企业级应用
  • 需要清晰职责分离的项目
  • 团队规模中等（10-50人）

3. 微服务架构 (Microservices Architecture)

核心思想

将大型应用拆分为一组小型、独立的服务，每个服务围绕业务能力构建。

关键特征

  • 服务自治：独立开发、部署、扩展
  • 技术多样性：每个服务可用不同技术栈
  • 去中心化治理：团队对服务全权负责
  • 智能端点，哑管道：服务包含业务逻辑，通信简单

优点

  • ✅ 独立部署：服务可单独部署，降低风险
  • ✅ 技术自由：团队可选择合适的技术栈
  • ✅ 弹性扩展：可按需扩展特定服务
  • ✅ 容错性：一个服务故障不影响整体

缺点

  • ❌ 分布式复杂性：网络延迟、服务发现、配置管理
  • ❌ 数据一致性：分布式事务复杂
  • ❌ 运维复杂：需要完善的监控、日志、告警
  • ❌ 团队协作成本：跨团队沟通成本高

适用场景

  • 大型复杂系统
  • 需要快速迭代的业务
  • 多团队协作（>50人）
  • 高可用性要求

4. 事件驱动架构 (Event-Driven Architecture)

核心思想

组件通过事件进行异步通信，实现松耦合。

核心概念

  • 事件：系统中发生的重要事情
  • 事件生产者：产生事件的组件
  • 事件消费者：处理事件的组件
  • 事件总线/消息队列：事件传递的媒介

常见模式

  • 发布-订阅 (Pub-Sub)
  • 事件溯源 (Event Sourcing)
  • CQRS (Command Query Responsibility Segregation)

优点

  • ✅ 松耦合：组件间依赖少
  • ✅ 高扩展性：易于水平扩展
  • ✅ 实时性：支持实时数据处理
  • ✅ 弹性：消费者故障不影响生产者

缺点

  • ❌ 事件顺序：保证事件顺序复杂
  • ❌ 调试困难：异步流程难以追踪
  • ❌ 数据一致性：最终一致性，非强一致性
  • ❌ 复杂度：需要处理消息丢失、重复等问题

适用场景

  • 实时数据处理
  • 异步业务流程
  • 需要高扩展性的系统
  • 微服务间的通信

5. 六边形架构 (Hexagonal Architecture)

核心思想

将核心业务逻辑与外部依赖隔离，通过端口和适配器与外部世界交互。

核心概念

  • 领域层：核心业务逻辑
  • 端口：领域层定义的接口
  • 适配器：实现端口的具体技术实现
  • 外部世界：数据库、UI、外部服务等

优点

  • ✅ 可测试性：核心逻辑可独立测试
  • ✅ 可替换性：外部依赖易于替换
  • ✅ 业务聚焦：核心业务逻辑清晰
  • ✅ 技术无关：领域层不依赖具体技术

缺点

  • ❌ 实现复杂：需要更多抽象层
  • ❌ 学习曲线：概念较新，团队需要适应
  • ❌ 过度设计风险：可能增加不必要的复杂性

适用场景

  • 领域驱动设计 (DDD) 项目
  • 需要长期维护的核心系统
  • 技术栈可能变化的项目

6. 服务网格 (Service Mesh)

核心思想

在应用代码之外，为服务间通信提供统一的基础设施层。

核心组件

  • 数据平面：处理服务间通信（如 Envoy）
  • 控制平面：管理配置和策略（如 Istio）
  • Sidecar 代理：每个服务实例旁运行的代理

核心功能

  • 流量管理：负载均衡、路由、熔断
  • 安全：mTLS、认证、授权
  • 可观测性：指标、日志、追踪
  • 策略执行：速率限制、访问控制

优点

  • ✅ 透明性：应用代码无需修改
  • ✅ 统一管理：所有服务通信统一管理
  • ✅ 功能丰富：提供完整的微服务治理能力
  • ✅ 语言无关：支持多种编程语言

缺点

  • ❌ 运维复杂度：需要专业运维知识
  • ❌ 性能开销：Sidecar 增加延迟
  • ❌ 学习曲线：概念和工具较新
  • ❌ 资源消耗：每个实例需要额外资源

适用场景

  • 大规模微服务部署
  • 多语言技术栈
  • 需要统一治理的平台
  • 云原生应用

架构选择指南

选择原则

  1. 从简单开始：除非明确需要，否则从单体开始
  2. 渐进式演进：根据业务需求逐步调整架构
  3. 团队能力：选择团队能驾驭的架构
  4. 业务需求：架构服务于业务，不是炫技

决策矩阵

场景 推荐架构 理由
初创 MVP 单体架构 快速验证，成本低
企业应用 分层架构 职责清晰，易于维护
电商平台 微服务 独立扩展，高可用
实时系统 事件驱动 异步处理，高并发
核心系统 六边形架构 业务聚焦，长期维护
大规模微服务 服务网格 统一治理，多语言支持

演进路径建议

总结

架构模式没有绝对的好坏，只有适合与否。选择架构时需要考虑：

  1. 业务需求：系统要解决什么问题？
  2. 团队规模：有多少人开发和维护？
  3. 技术能力：团队熟悉哪些技术？
  4. 演进路线：未来可能如何发展？

记住：好的架构是演进而来的，不是设计出来的。从简单开始，根据实际需求逐步调整，才是架构设计的正确姿势。

架构设计的基本原则

发表于 2026-02-22 更新于 2026-02-24 分类于 架构基础

引言

在上一篇文章中，我们建立了架构思维的基础，理解了架构的价值。现在，让我们深入探讨架构设计的核心原则。这些原则是优秀架构的基石，无论技术如何变化，这些基本原则始终适用。

本文将介绍架构设计中最重要、最实用的基本原则，帮助你建立坚实的设计基础。

一、SOLID原则：面向对象设计的基石

SOLID原则是Robert C. Martin提出的五个面向对象设计原则的缩写，它们同样适用于架构设计。

1.1 单一职责原则 (Single Responsibility Principle, SRP)

一个类（或模块）应该只有一个引起变化的原因。

实践建议：

  • 每个模块只负责一个明确的业务功能
  • 避免”上帝类”或”万能模块”
  • 通过职责划分提高可维护性

1.2 开闭原则 (Open-Closed Principle, OCP)

软件实体应该对扩展开放，对修改关闭。

实践建议：

  • 通过接口和抽象实现扩展性
  • 使用策略模式、工厂模式等设计模式
  • 避免直接修改现有代码

1.3 里氏替换原则 (Liskov Substitution Principle, LSP)

子类必须能够替换其父类，而不影响程序的正确性。

实践建议：

  • 子类不应该改变父类的行为约定
  • 避免在子类中抛出父类没有声明的异常
  • 保持接口契约的一致性

1.4 接口隔离原则 (Interface Segregation Principle, ISP)

客户端不应该依赖它不需要的接口。

实践建议：

  • 创建小而专一的接口
  • 避免”胖接口”
  • 按需实现接口

1.5 依赖倒置原则 (Dependency Inversion Principle, DIP)

高层模块不应该依赖低层模块，两者都应该依赖抽象。

实践建议：

  • 依赖抽象，而不是具体实现
  • 使用依赖注入
  • 提高代码的可测试性和灵活性

二、DRY原则：避免重复

Don’t Repeat Yourself (不要重复你自己)

2.1 DRY的核心思想

  • 每一处知识在系统中都应该有单一、明确、权威的表示
  • 重复是万恶之源，会导致维护困难

2.2 如何实践DRY原则

  1. 提取公共方法：将重复代码提取为函数或方法
  2. 使用模板：对于相似的代码结构
  3. 创建基类：对于共享的行为
  4. 配置化：将硬编码的值提取为配置

2.3 DRY的误区

  • 不要过度抽象：过早的抽象可能带来不必要的复杂度
  • 区分逻辑重复和偶然重复：有些重复是必要的

三、KISS原则：保持简单

Keep It Simple, Stupid (保持简单，傻瓜)

3.1 简单性的价值

  • 简单的设计更容易理解和维护
  • 减少出错的可能性
  • 提高开发效率

3.2 如何实践KISS原则

  1. 避免过度设计：只解决当前的问题
  2. 选择简单的解决方案：在满足需求的前提下选择最简单的
  3. 清晰的命名：让代码自解释
  4. 减少依赖：最小化外部依赖

四、YAGNI原则：你不需要它

You Aren’t Gonna Need It (你不需要它)

4.1 YAGNI的核心

  • 不要为未来可能的需求编写代码
  • 专注于当前的需求
  • 避免”以防万一”的设计

4.2 如何实践YAGNI

  1. 按需开发：只在需要时添加功能
  2. 重构优于预测：当需求变化时重构代码
  3. 保持代码灵活：但不为不存在的需求设计

五、关注点分离 (Separation of Concerns, SoC)

5.1 SoC的核心思想

  • 将系统分解为不同的部分，每个部分解决一个特定的关注点
  • 降低系统的复杂度

5.2 实践示例

  • MVC模式：模型、视图、控制器分离
  • 分层架构：表现层、业务层、数据层分离
  • 微服务：按业务领域分离服务

六、其他重要原则

6.1 最小惊讶原则 (Principle of Least Astonishment)

  • 系统的行为应该符合用户的预期
  • 避免令人惊讶的设计

6.2 高内聚低耦合 (High Cohesion, Low Coupling)

  • 高内聚：模块内部元素紧密相关
  • 低耦合：模块之间依赖最小化

6.3 失败快速原则 (Fail Fast)

  • 尽早发现并报告错误
  • 避免错误在系统中传播

七、原则的权衡与平衡

7.1 没有银弹

  • 不同原则之间可能存在冲突
  • 需要根据具体场景权衡

7.2 平衡的艺术

场景 优先考虑的原则
快速原型 KISS, YAGNI
长期维护的系统 SOLID, DRY
高性能系统 简单性，避免过度抽象
团队协作项目 一致性，清晰的接口

八、实践指南

8.1 代码审查清单

  • 每个类/模块是否只有一个主要职责？
  • 是否有重复的代码可以提取？
  • 设计是否足够简单？
  • 是否添加了不必要的功能？
  • 关注点是否清晰分离？

8.2 重构时机

  1. 发现重复代码时立即重构
  2. 添加新功能困难时考虑设计问题
  3. 代码难以理解时简化设计
  4. 测试困难时检查耦合度

总结

架构设计原则是指南针，不是铁律。理解这些原则的核心思想比机械地应用它们更重要。在实际工作中，你需要：

  1. 理解原则背后的思想
  2. 根据具体场景灵活应用
  3. 在原则之间找到平衡
  4. 持续学习和反思

记住：好的架构是演进而来的，不是设计出来的。从简单的设计开始，根据需求逐步演进。

思考与实践

思考题

  1. 在你的项目中，哪个原则被违反得最多？造成了什么后果？
  2. SOLID原则中，你认为哪个原则最重要？为什么？
  3. 什么时候应该违反这些原则？

实践任务

  1. 选择一个你熟悉的项目，用本文的原则分析其设计
  2. 找出至少3处可以改进的地方
  3. 制定一个简单的重构计划
